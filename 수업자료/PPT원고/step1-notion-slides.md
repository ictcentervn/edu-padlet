## 슬라이드 2: 생각해보기 — 카카오톡 데이터는 어디에?

**핵심 텍스트:**
- 카카오톡 메시지를 보낸 뒤, 폰을 껐다 켜도 남아 있다
- 이 메시지는 어디에 저장되어 있을까?
- 인스타 사진은? 유튜브 댓글은?

**시각 자료 제안:** 카카오톡 채팅 화면 일러스트 + 물음표 아이콘 + 화살표가 "???" 를 가리킴

**선생님 멘트:** "핸드폰을 껐다 켜도 메시지가 남아 있죠? 그럼 이 데이터는 내 핸드폰 안에만 있는 걸까요, 아니면 어딘가 다른 곳에도 저장되어 있을까요? 손 들어서 이야기해 봅시다."

## 슬라이드 3: 정답 — 서버 안의 데이터베이스!

**핵심 텍스트:**
- 정답: 서버 안에 있는 **데이터베이스**에 저장
- 카카오톡, 인스타, 유튜브 전부 마찬가지
- 오늘 우리도 데이터를 저장하는 앱을 직접 만든다!

**시각 자료 제안:** 스마트폰 -> 인터넷 구름 -> 서버 건물 -> DB 아이콘 (흐름 화살표)

**선생님 멘트:** "전부 서버에 있는 데이터베이스에 저장되어 있습니다. 오늘 우리도 이런 데이터베이스를 직접 연결해서, 카드를 저장하는 앱을 만들어 볼 거예요!"

---
## 슬라이드 4: 데이터베이스란? — 도서관 장부 비유

**핵심 텍스트:**
- 데이터베이스(DB) = 데이터를 정리해서 보관하는 장부
- 한 줄(행) = 데이터 하나 (레코드)
- 한 칸(열) = 데이터의 종류 (컬럼)
- 우리 앱에서는 카드 하나 = 장부의 한 줄

**시각 자료 제안:** 도서관 대출 장부 표 (학생 이름 / 빌린 책 / 날짜 / 반납 여부) + 옆에 우리 앱 카드 데이터 표 (제목 / 내용 / 색상)를 나란히 배치

**선생님 멘트:** "도서관 대출 장부를 생각해 보세요. 누가 어떤 책을 언제 빌렸는지 표로 정리하죠? 데이터베이스가 바로 이 장부입니다. 우리 앱에서는 카드 하나하나가 장부의 한 줄이 됩니다."

---
## 슬라이드 5: 서버란? — 식당 주방 비유

**핵심 텍스트:**
- 손님(브라우저)이 주문(요청)을 한다
- 주방(서버)이 요리(처리)를 한다
- 완성된 요리(응답)를 돌려준다
- 우리 앱의 서버 = server.js (Express)

**시각 자료 제안:** 왼쪽: 손님 아이콘(브라우저) -> 가운데: 주방 아이콘(서버) -> 오른쪽: 음식(응답). 아래에 "비빔밥 주세요" / "비빔밥 나왔습니다" 말풍선

**선생님 멘트:** "식당에서 주문하면 주방에서 요리를 만들어 주죠? 서버도 똑같습니다. 브라우저가 '카드 보여줘'라고 요청하면, 서버가 데이터베이스에서 카드를 꺼내서 보내줍니다."

---
## 슬라이드 6: API란? — 주문서 비유

**핵심 텍스트:**
- API = 정해진 형식의 주문서
- 메뉴에 없는 건 주문할 수 없다
- 우리 앱의 API 4가지:
  - GET /api/cards (카드 목록)
  - POST /api/cards (카드 만들기)
  - PATCH /api/cards/1 (카드 수정)
  - DELETE /api/cards/1 (카드 삭제)

**시각 자료 제안:** 메뉴판 모양의 카드에 4가지 API가 적힌 다이어그램. 각 API 옆에 아이콘 (목록/추가/수정/삭제)

**선생님 멘트:** "식당에서 아무 말이나 하면 안 되죠. 메뉴판에 있는 것만 주문할 수 있습니다. API도 마찬가지예요. 정해진 양식대로 요청해야 서버가 알아듣습니다."

---
## 슬라이드 7: 우리 앱 전체 구조도

**핵심 텍스트:**
- 브라우저 -> Express 서버 -> Notion DB
- 브라우저가 서버에 요청
- 서버가 Notion API로 데이터 가져옴
- 서버가 브라우저에 응답

**시각 자료 제안:** 3단 흐름도 — [브라우저 HTML/CSS/JS] --(HTTP 요청)--> [Express 서버 server.js] --(Notion API)--> [Notion DB 클라우드]. 화살표 위에 "fetch", "notion.databases.query" 등 라벨

**선생님 멘트:** "전체 구조를 보면 이렇습니다. 브라우저가 서버한테 부탁하고, 서버가 Notion한테 데이터를 가져와서 다시 브라우저에 전해줍니다. 중간에 서버가 있는 이유는, 브라우저가 직접 Notion에 접근하면 보안 문제가 생기기 때문이에요."

---
## 슬라이드 8: Notion API 키 발급 (설정 안내)

**핵심 텍스트:**
- Notion Integration 생성 (API 키 발급)
- Notion 데이터베이스 만들기 (제목/내용/색상 컬럼)
- 데이터베이스에 Integration 연결
- .env 파일에 API 키와 DB ID 저장

**시각 자료 제안:** 4단계 번호가 매겨진 스크린샷 위치 표시 (1. Notion Integration 페이지 2. 키 복사 화면 3. DB 공유 메뉴 4. .env 파일). 실제 스크린샷은 실습 교안 참고

**선생님 멘트:** "지금부터 Notion API 키를 발급받겠습니다. 화면을 따라 하세요. 잘 안 되는 부분은 손 들어 주세요. 실습 교안에 스크린샷이 있으니 참고하세요."

## 슬라이드 12: Notion에서 데이터 확인 (Wow 순간!)

**핵심 텍스트:**
- 브라우저에서 추가한 카드가 Notion DB에 저장됨!
- Notion을 열어보면 데이터가 실시간으로 들어와 있음
- 내가 만든 앱이 진짜 데이터베이스에 연결된 것!

**시각 자료 제안:** 왼쪽: 우리 앱 화면(카드 3개) / 오른쪽: Notion DB 화면(같은 데이터 3줄). 가운데에 "=" 또는 "연결됨!" 표시

**선생님 멘트:** "자, 이제 Notion을 열어 보세요! 여러분이 방금 추가한 카드가 여기에도 보이죠? 이게 오늘의 핵심입니다. 브라우저에서 입력한 데이터가 Notion 데이터베이스에 저장된 거예요. 신기하지 않나요?"

## 슬라이드 13: 오늘 배운 것 정리

**핵심 텍스트:**

**시각 자료 제안:** 4개 키워드를 카드 형태로 배치. 각각에 아이콘 (장부/주방/주문서/구조도)

**선생님 멘트:** "오늘 3가지 핵심 개념을 배웠습니다. 데이터베이스, 서버, API. 그리고 이 세 가지가 어떻게 연결되는지 직접 체험했어요. 이제 여러분은 실제 데이터베이스에 연결된 앱을 만든 경험이 있는 사람입니다!"
