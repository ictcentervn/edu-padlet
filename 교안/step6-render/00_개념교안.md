# Render — 실제 서비스 배포와 CI/CD 입문

> **대상:** 고등학교 2학년 ~ 3학년 심화 (Git/GitHub 사용 경험 필수, Step 4 완료 전제)

---

## 오늘 배울 내용

> GitHub에 코드를 push하면 **자동으로 서버에 배포**됩니다. 실제 개발자들이 매일 일하는 방식 그대로, 우리 패들렛 앱을 세상에 공개합니다.

---

## 생각해보기

프로그래머가 코드를 고칠 때마다 서버에 직접 들어가서 파일을 올릴까요?

- 코드를 수정했으니 서버 컴퓨터에 USB로 복사?
- 수정할 때마다 서버에 원격 접속해서 파일 교체?
- 100명이 함께 개발하는 회사에서 이런 식으로 할 수 있을까요?

**아니요!** 코드를 push하면 **자동으로 배포**됩니다.

이 과정을 **CI/CD**라고 합니다. 오늘 우리도 직접 체험합니다!

---

## 핵심 개념 1: 배포 (Deploy)

```
   ┌──────────┐                    ┌──────────────────────┐
   │  내 컴퓨터 │   ── 배포 ──▶    │  서버 (인터넷에 공개)   │
   │          │                    │                      │
   │ 코드 작성  │                    │  누구나 접속 가능!     │
   │ 테스트     │                    │  https://내앱.com     │
   └──────────┘                    └──────────────────────┘
```

### 비유로 이해하기: 원고를 출판사에 보내기

- 내 컴퓨터에서 코딩하는 것 = **집에서 원고 쓰기**
- 서버에 배포하는 것 = **출판사에 원고를 보내서 책으로 만드는 것**
- 배포가 완료되면 = **서점(인터넷)에서 누구나 읽을 수 있음**

Codespaces에서도 배포를 해봤습니다. 하지만 Codespaces는 "코딩 + 배포"가 한 곳에 있었죠.

오늘 배울 **Render**는 **GitHub에서 코드를 가져와서 자동으로 배포**해줍니다.

---

## 핵심 개념 2: CI/CD — 자동 배포의 핵심

```
CI/CD = Continuous Integration / Continuous Deployment
        (지속적 통합)            (지속적 배포)
```

쉽게 말하면:

```
코드를 push하면 → 알아서 테스트하고 → 알아서 배포한다
```

### 비유로 이해하기: 출판사의 자동 인쇄 시스템

```
[수동 방식 — 예전]
작가가 원고 수정 → 출판사에 전화 → 편집자가 확인 → 인쇄소에 전달 → 새 판 출간
                   느리고 번거로움!

[자동 방식 — CI/CD]
작가가 원고 수정 → 클라우드에 업로드 → 자동 맞춤법 검사 → 자동 인쇄 → 새 판 출간
                   빠르고 자동!
```

우리가 할 과정:

```
코드 수정 → git push → GitHub → Render가 감지
                                      ↓
                                 자동 빌드 (npm install)
                                      ↓
                                 자동 시작 (node server.js)
                                      ↓
                                 https://내앱.onrender.com 업데이트!
```

**git push 한 번이면 끝.** 나머지는 전부 자동입니다.

## 핵심 개념 3: 환경변수 — 비밀 설정의 클라우드 버전

Step 3에서 `.env` 파일에 Supabase 키를 저장했던 것, 기억나나요?

```
# .env 파일 (내 컴퓨터)
SUPABASE_URL=https://xxx.supabase.co
SUPABASE_ANON_KEY=eyJhbG...
```

하지만 `.env` 파일은 GitHub에 올리면 안 됩니다 (Step 4에서 `.gitignore`로 막았죠).

그러면 서버에서는 이 값을 어떻게 알까요?

**Render 대시보드에서 직접 입력합니다.**

```
┌─────────────────────────────────────────────┐
│  Render 대시보드 → Environment 탭             │
│                                             │
│  Key                │ Value                 │
│  ───────────────────┼─────────────────────  │
│  SUPABASE_URL       │ https://xxx.supabase  │
│  SUPABASE_ANON_KEY  │ eyJhbG...             │
│                                             │
│  [Add Environment Variable]                 │
└─────────────────────────────────────────────┘
```

이렇게 하면:
- 비밀 키가 코드에 포함되지 않아 **안전**합니다
- 서버에서 `process.env.SUPABASE_URL`로 읽을 수 있습니다
- `.env` 파일과 같은 역할이지만, **클라우드에서 관리**합니다

## Codespaces vs Render 비교

| 항목 | Codespaces | Render |
|------|--------|--------|
| **코딩 방식** | 브라우저에서 직접 코딩 | 로컬(VS Code)에서 코딩 후 push |
| **배포 방식** | 코드 수정하면 자동 반영 | GitHub push하면 자동 배포 |
| **코드 관리** | Codespaces 자체 저장소 | GitHub 저장소 |
| **환경변수** | .env 편집기 제공 | 대시보드에서 설정 |
| **무료 슬립** | 5분 비활동 시 슬립 | 15분 비활동 시 슬립 |
| **적합한 용도** | 교육, 프로토타입 | 실무, 실제 서비스 |
| **비유** | 연습장 | 출판사 |

> **Codespaces는 연습장**이고, **Render는 출판사**입니다.
> 연습장에서 초안을 쓰고, 출판사에서 책을 만들어 세상에 공개하는 것처럼요.

## 전체 배포 흐름 — 한눈에 보기

```
[내 컴퓨터 — VS Code]
        │
        │  git push origin main
        ▼
[GitHub 저장소]
        │
        │  Render가 변경을 감지 (Webhook)
        ▼
[Render 서버]
        │
        ├─ 1. 코드 다운로드 (git clone)
        ├─ 2. npm install (패키지 설치)
        ├─ 3. node server.js (서버 시작)
        │
        ▼
[https://내앱.onrender.com]
        │
        ▼
  전 세계 누구나 접속 가능!
```

---

## 6단계 전체 여정 — 우리가 걸어온 길

```
Step 1: DB 개념 (Notion)      ← "데이터를 어디에 저장하지?"
   ↓
Step 2: SQL 입문 (SQLite)     ← "DB를 직접 다뤄보자"
   ↓
Step 3: 클라우드 DB (Supabase) ← "인터넷에서 DB를 쓰자"
   ↓
Step 4: 코드 관리 (GitHub)     ← "코드를 안전하게 관리하자"
   ↓
Step 5: 즉시 배포 (Codespaces)    ← "브라우저로 코딩하고 바로 공유"
   ↓
Step 6: 실무 배포 (Render)    ← 지금 여기! "진짜 개발자처럼 배포"
```

Step 1에서 시작한 작은 패들렛 앱이, 6단계를 거쳐 **전 세계 누구나 접속할 수 있는 실제 서비스**가 됩니다.

---

## 이번 단계에서 바뀐 것

| 항목 | 변경 내용 |
|------|---------|
| server.js | PORT 설정 확인 (Step 5에서 이미 수정했다면 변경 없음) |
| db.js | 변경 없음 |
| public/ | 변경 없음 |
| .env | Render 대시보드에서 환경변수 입력 (코드에는 없음) |
| package.json | 변경 없음 |

**코드는 거의 그대로입니다.** 바뀌는 것은 "어디서 배포하느냐"와 "어떻게 배포하느냐"뿐입니다.

> 비유: 같은 원고(코드)를 출판사(Codespaces)에서 다른 출판사(Render)로 옮기는 것과 같습니다. 원고는 그대로이고, 출판 방식만 달라집니다.

## 학습 목표 체크리스트

### 🟢 입문 (기본 따라하기)
- [ ] 배포(Deploy)가 무엇인지 한 문장으로 설명할 수 있다
- [ ] Render에서 앱을 배포하고 URL로 접속할 수 있다
- [ ] 환경변수를 Render 대시보드에서 설정할 수 있다
- [ ] 배포된 앱의 URL을 친구에게 공유할 수 있다

### 🟡 기본 (원리 이해)
- [ ] CI/CD가 무엇인지 비유로 설명할 수 있다
- [ ] Codespaces와 Render의 차이를 3가지 이상 설명할 수 있다
- [ ] git push → 자동 배포 흐름을 단계별로 설명할 수 있다
- [ ] Render 로그를 확인해서 빌드 상태를 파악할 수 있다

### 🔴 심화 (실무 도전)
- [ ] 코드를 수정하고 push해서 자동 배포를 직접 체험할 수 있다
- [ ] 배포 실패 시 로그를 보고 원인을 찾아 해결할 수 있다
- [ ] 6단계 전체 여정을 정리하고 각 단계의 역할을 설명할 수 있다
