# 자주 막히는 문제 해결법 — Step 6 (Render)

## 🔴 1. 빌드 실패 — "npm install" 에러

### 증상
Render 빌드 로그에 빨간 글씨로 에러가 나타나고, 배포가 중단됩니다.

```
==> Running build command 'npm install'
npm ERR! ...
==> Build failed!
```

### 원인별 해결법

**"npm ERR! missing script: start"**
→ package.json에 start 스크립트가 없는 경우. 하지만 Render에서 Start Command를 직접 설정했다면 이 에러는 나지 않습니다.

**"npm ERR! code ERESOLVE" 또는 의존성 충돌**
→ package-lock.json이 오래되었거나 충돌이 있는 경우
```bash
# 로컬에서 실행 후 다시 push
rm -rf node_modules package-lock.json
npm install
git add package-lock.json
git commit -m "package-lock 재생성"
git push origin main
```

**package.json이 루트 폴더에 없음**
→ Render는 저장소 루트에서 `npm install`을 실행합니다. package.json이 하위 폴더에 있다면 Render 설정에서 "Root Directory"를 지정하세요.

## 🔴 2. 서버 시작 실패 — Start Command 오류

### 증상
빌드는 성공했지만 서버가 시작되지 않습니다.

```
==> Build successful!
==> Starting service with 'node server.js'
Error: Cannot find module '/opt/render/project/src/server.js'
```

### 해결법

**파일 이름 확인**
→ Start Command에 입력한 파일 이름이 실제 파일 이름과 일치하는지 확인
```
Start Command: node server.js  ← 파일명이 Server.js(대문자)면 안 됩니다
```

**파일 위치 확인**
→ server.js가 저장소 루트에 있는지 확인. 하위 폴더에 있다면 경로를 포함:
```
Start Command: node src/server.js
```

## 🔴 3. 환경변수 누락

### 증상
서버는 시작되지만, 앱이 제대로 동작하지 않습니다. 카드가 로드되지 않거나 "undefined" 에러가 발생합니다.

```
Error: supabaseUrl is required
```
또는
```
TypeError: Cannot read properties of undefined
```

### 해결법

1. Render 대시보드 → 내 서비스 → **"Environment"** 탭
2. 아래 두 변수가 있는지 확인:
   - `SUPABASE_URL` — 값이 `https://`로 시작하는지 확인
   - `SUPABASE_ANON_KEY` — 값이 `eyJ`로 시작하는지 확인
3. 값을 복사-붙여넣기 할 때 앞뒤 공백이 들어가지 않았는지 확인
4. 환경변수를 추가/수정한 후 **서비스를 재배포** (Manual Deploy → Deploy latest commit)

> 💡 가장 흔한 실수: 환경변수 Key에 오타 (`SUPABASE_ANON_KEY`를 `SUPABASE_KEY`로 쓰는 등)

## 🟡 4. "Application error" 화면

### 증상
배포된 URL에 접속하면 "Application error" 또는 "This page isn't working" 화면이 나타납니다.

### 해결법

1. Render 대시보드 → 내 서비스 → **"Logs"** 탭 클릭
2. 로그에서 에러 메시지를 확인
3. 에러 내용에 따라 대응:

| 로그 내용 | 원인 | 해결 |
|-----------|------|------|
| `Cannot find module 'express'` | npm install 실패 | Build Command가 `npm install`인지 확인 |
| `supabaseUrl is required` | 환경변수 누락 | #3 참조 |
| `EADDRINUSE` | 포트 충돌 | process.env.PORT 사용 확인 (#8 참조) |
| `SyntaxError` | 코드 문법 에러 | 로컬에서 `node server.js` 실행해서 확인 |

## 🟡 5. 무료 슬립 — 접속이 느림

### 증상
앱 URL에 접속하면 30초~1분 동안 로딩만 됩니다.

### 원인
Render 무료 플랜은 **15분 동안 아무도 접속하지 않으면** 서버를 슬립 모드로 전환합니다.
누군가 접속하면 다시 깨어나는데, 이때 30초~1분이 걸립니다.

### 해결법
- 이것은 정상 동작입니다. 기다리면 됩니다.
- 수업 시연 전에 미리 한 번 접속해서 서버를 깨워두세요.
- 유료 플랜으로 업그레이드하면 슬립이 없어집니다 (수업에서는 불필요).

## 🟡 6. GitHub 연결 안 됨

### 증상
Render에서 "Connect a repository" 화면에 내 GitHub 저장소가 보이지 않습니다.

### 해결법

1. **"Configure account"** 또는 **"+ Connect another GitHub account"** 클릭
2. GitHub으로 이동 → Render 앱 설치 화면
3. **"All repositories"** 선택하거나, 특정 저장소(edu-padlet)를 선택
4. **"Install"** 클릭
5. Render로 돌아가서 새로고침 → 저장소가 목록에 나타남

> GitHub에 로그인된 계정이 맞는지도 확인하세요. 다른 계정으로 로그인되어 있을 수 있습니다.

## 🟡 7. 자동 배포가 안 됨

### 증상
`git push`를 했는데 Render에서 새 배포가 시작되지 않습니다.

### 확인 순서

1. **올바른 브랜치에 push했나요?**
   ```bash
   git branch     # 현재 브랜치 확인
   git push origin main   # main 브랜치에 push
   ```
   Render에 설정한 브랜치(보통 `main`)에 push해야 합니다.

2. **Auto-Deploy가 켜져 있나요?**
   → Render 대시보드 → Settings → Build & Deploy → Auto-Deploy를 `Yes`로 설정

3. **push가 실제로 GitHub에 반영되었나요?**
   → GitHub.com에서 저장소를 확인. 최신 커밋이 보이는지 확인.

4. **수동 배포로 확인:**
   → Render 대시보드 → "Manual Deploy" → "Deploy latest commit" 클릭

## 🔴 8. PORT 에러 — 서버 시작 실패

### 증상
```
Error: listen EACCES: permission denied 0.0.0.0:80
```
또는 서버가 시작되었다고 나오지만 접속이 안 됩니다.

### 원인
Render는 `PORT` 환경변수를 자동으로 제공합니다. 이 값을 사용하지 않으면 배포가 실패합니다.

### 해결법

**server.js 확인:**
```javascript
// ✅ 올바른 코드
const PORT = process.env.PORT || 3000;
app.listen(PORT, () => {
  console.log(`✅ 서버 시작! 포트: ${PORT}`);
});
```

```javascript
// ❌ 틀린 코드 — Render에서 작동 안 함
app.listen(3000, () => {
  console.log('서버 시작!');
});
```

수정 후:
```bash
git add server.js
git commit -m "PORT 환경변수 사용"
git push origin main
```

## 🟡 9. 빌드 시간이 너무 오래 걸림

### 증상
빌드가 5분 이상 걸리거나, 10분 넘게 진행됩니다.

### 원인
- Render 무료 플랜은 빌드 서버 성능이 제한적입니다.
- 패키지가 많으면 설치 시간이 길어집니다.

### 해결법
- **2~5분은 정상입니다.** 기다리세요.
- 10분 이상 걸리면 빌드 로그를 확인. 특정 패키지에서 멈춰있다면 해당 패키지 문제일 수 있습니다.
- 불필요한 패키지가 package.json에 있다면 제거하세요.
- 무료 플랜의 한계이므로, 수업 중에는 빌드 대기 시간에 다른 활동을 하세요.

## 🔴 10. 커스텀 도메인 설정 안 됨

### 증상
커스텀 도메인을 추가했지만 접속이 안 됩니다.

### 해결법 (심화)

1. Render 대시보드 → Settings → Custom Domains에서 도메인 추가
2. Render가 안내하는 **DNS 레코드**(CNAME)를 도메인 관리자 사이트에서 설정
3. DNS 전파에 최대 24~48시간 소요
4. HTTPS는 Render가 자동으로 설정 (Let's Encrypt)

> 이 단계는 도메인을 소유하고 있는 학생만 가능합니다. 수업에서는 기본 `.onrender.com` 도메인으로 충분합니다.

## 6단계 전체 트러블슈팅 비교표

6단계를 모두 거치면서 각 단계에서 가장 흔했던 에러를 정리합니다:

| 단계 | 도구 | 가장 흔한 에러 Top 1 | 한 줄 해결법 |
|------|------|-------------------|------------|
| Step 1 | Notion | API 키 발급/연결 문제 | Integration을 DB에 연결했는지 확인 |
| Step 2 | SQLite | better-sqlite3 빌드 에러 | Python, C++ 빌드 도구 설치 필요 |
| Step 3 | Supabase | Data API 위치 찾기 | Settings → API에서 URL과 Key 확인 |
| Step 4 | GitHub | .env 실수로 올림 | .gitignore에 .env 추가, git rm --cached |
| Step 5 | Glitch | PORT 환경변수 | process.env.PORT 사용 |
| Step 6 | Render | 환경변수 누락 | 대시보드 Environment 탭에서 설정 |

### 공통 해결 전략

어떤 단계에서든 에러가 나면 이 순서로 확인하세요:

```
1. 에러 메시지를 정확히 읽는다 (빨간 글씨를 무시하지 마세요!)
2. 에러 메시지를 검색한다 (Google 또는 AI에게 물어보기)
3. 환경변수를 확인한다 (대부분의 에러 원인)
4. 로그를 확인한다 (Render Logs, 브라우저 Console, 터미널)
5. 코드를 로컬에서 먼저 테스트한다 (node server.js)
```

> 💡 교사 메모: 학생들이 에러 메시지를 읽지 않고 "안 돼요"라고만 하는 경우가 많습니다. "에러 메시지를 소리 내서 읽어보세요"라고 안내하면 효과적입니다. 에러 메시지 읽기는 개발자의 가장 중요한 습관입니다.
