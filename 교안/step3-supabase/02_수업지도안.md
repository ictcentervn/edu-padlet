> **대상:** 고등학교 2~3학년 (심화)
> **시간:** 2차시 x 45분 (총 90분)
> **전제 조건:** Step 1(Notion) + Step 2(SQLite) 수업 완료
> **준비물:** 노트북(충전 완료), VS Code 설치, Node.js 설치, GitHub 계정, 인터넷 연결

## 차시별 계획

---

### 1차시 — 클라우드 DB 개념 + Supabase 설정 + Step 2→3 전환 (45분)

**학습 목표:** 클라우드 DB의 개념을 이해하고, Supabase를 설정하여 Step 2 앱을 Step 3으로 전환한다.

| 시간 | 활동 | 교사 행동 | 비고 |
|------|------|---------|------|
| 0~3분 | **도입:** "padlet.db를 친구에게 보내면 친구도 쓸 수 있을까?" | 생각해보기 질문 | 답: 안 됨! 내 컴퓨터에만 있으니까 |
| 3~8분 | **개념 설명:** 3단계 전체 비교 다이어그램 | 화면 공유, 3단계 여정 설명 | 00_개념교안.md 활용 |
| 8~13분 | **개념 설명:** 클라우드 DB란? Supabase란? | 비교표 보여주기 | "Notion처럼 웹에서 확인 가능!" |
| 13~15분 | **전환:** "server.js와 화면은 역시 안 바뀌었어요!" | 3번째로 같은 패턴 강조 | 모듈화의 힘 |
| 15~30분 | **실습:** Supabase 프로젝트 만들기 | 교사 화면 공유하며 같이 진행 | 01_Supabase설정가이드.md 1~5단계 |
| 30~42분 | **실습:** 프로젝트 준비 + .env 설정 + 서버 실행 | 돌아다니며 개별 지원 | 03_학생실습지침서.md A~C단계 |
| 42~45분 | **확인:** 카드 추가 → Supabase 대시보드에서 확인 | "Notion에서 확인했던 것처럼!" | 핵심 교육 포인트! |

> 💡 **교사 메모:**
> - Supabase 프로젝트 생성 시 1~2분 대기 시간이 있습니다. 이 시간에 개념 보충 설명을 하세요.
> - SQL Editor에서 SQL 붙여넣기 + Run은 학생들이 빠르게 할 수 있습니다. Step 2에서 SQL을 배웠기 때문입니다.
> - **핵심 순간:** 카드를 추가한 후 Supabase Table Editor에서 데이터가 보이는 것을 확인시키세요. "Step 1에서 Notion 대시보드에서 확인한 것 기억나죠? 이번엔 Supabase에서 확인하는 거예요!"
> - URL/Key 복사에서 실수가 많습니다. 반드시 "복사 아이콘"을 클릭하도록 안내하세요.
> - **시간이 부족하면:** 개념 설명(0~15분)을 5분으로 압축하고, 바로 실습으로 넘어가세요.

### 2차시 — 동작 확인 + 3단계 전체 회고 + 선택 심화 (45분)

**학습 목표:** 클라우드 DB의 특성을 체험하고, 3단계 전체 학습을 회고한다.

| 시간 | 활동 | 교사 행동 | 비고 |
|------|------|---------|------|
| 0~3분 | **복습:** "Supabase에 데이터 저장한 거 확인해볼까?" | Table Editor 보여주기 | 1차시 연결 |
| 3~10분 | **실습:** 서버 껐다 켜도 데이터 유지 확인 | 시연 + 학생 실습 | "SQLite와 같은 결과!" |
| 10~18분 | **실습:** 두 브라우저 창 동시 접속 테스트 | "한쪽에서 추가, 다른 쪽에서 새로고침" | 클라우드 DB의 핵심! |
| 18~28분 | **3단계 전체 회고** | 비교표 보여주며 토론 | 아래 상세 설명 |
| 28~40분 | **선택 심화 활동** | 수준별 안내 | 아래 상세 설명 |
| 40~45분 | **정리:** 학습 목표 체크리스트 + 소감 | 00_개념교안.md 체크리스트 | "3단계 전체를 완주했어요!" |

> 💡 **교사 메모:**
> - **두 브라우저 동시 접속**이 이 수업의 하이라이트입니다! Chrome 창 2개를 열고, 한쪽에서 카드 추가 → 다른 쪽에서 새로고침 → 데이터가 보이는 것을 확인시키세요. "SQLite에서는 이게 안 됐죠? 클라우드니까 가능한 거예요!"
> - 3단계 회고는 학생들이 전체 여정을 정리하는 중요한 시간입니다. 급하게 넘기지 마세요.
> - 선택 심화(배포/Realtime)는 **시간이 부족하면 교사가 시연만** 해도 됩니다. "이런 것도 가능합니다" 정도로 보여주세요.

### 3단계 전체 회고 활동 (18~28분)

**진행 방법:**

1. **3단계 비교표 보여주기** (3분)
   - 00_개념교안.md의 비교표를 화면에 띄움
   - "Step 1에서 3까지 오면서 뭐가 바뀌고, 뭐가 안 바뀌었나요?"

2. **질문 던지기** (5분)
   - "세 단계 중 가장 어려웠던 건?" → 대부분 Step 1 (API 키!)
   - "세 단계 중 가장 실용적인 건?" → Step 3 (친구도 쓸 수 있으니까)
   - "server.js가 한 번도 안 바뀐 이유는?" → 모듈화!
   - "실제 서비스(인스타그램, 카카오톡)는 어떤 방식에 가까울까?" → Step 3!

3. **정리 발표** (2분)
   - 2~3명이 "3단계 중 가장 인상 깊었던 것" 발표

### 선택 심화 활동 옵션 (28~40분)

학생 수준에 따라 택 1:

**🟢 기본:** "나만의 보드 꾸미기"
- 카드 5개 이상 추가
- Supabase 대시보드에서 데이터 확인
- 서버 재시작 후 데이터 유지 확인

**🟡 중급:** 바이브코딩으로 기능 추가
- 04_바이브코딩프롬프트집.md의 중급 프롬프트 활용
- 실시간(Realtime) 기능 또는 인증(Auth) 기능 요청

**🔴 심화:** Vercel 배포 또는 Realtime 실시간
- Vercel로 앱을 인터넷에 배포하여 친구에게 링크 공유
- 또는 Supabase Realtime으로 실시간 협업 기능 추가
- 04_바이브코딩프롬프트집.md 심화 프롬프트 활용

> 💡 **교사 메모:** 배포(Vercel)와 Realtime은 시간이 부족하면 **교사가 시연만** 해도 충분합니다. "여러분이 만든 앱을 이렇게 인터넷에 올릴 수도 있어요!"라고 보여주는 것만으로도 학생들에게 큰 동기부여가 됩니다.

## 시간 조절 가이드

| 상황 | 대응 |
|------|------|
| **Step 2를 잘 했다면** | 1차시 개념 설명을 5분으로 줄이고 바로 실습 |

| **인터넷이 느리면** | 교사 화면으로 시연, 학생은 관찰 후 나중에 개별 실습 |
| **일부 학생만 빠른 경우** | 심화 활동 안내 (배포, Realtime, 바이브코딩) |

## Step 1, 2와 비교한 수업 설계 차이

| 항목 | Step 1 (3차시) | Step 2 (2차시) | Step 3 (2차시) |
|------|--------------|--------------|--------------|
| 개념 설명 시간 | 1차시 전체 | 1차시 전반부 | 1차시 10분 (이미 알고 있음) |
| 설정 시간 | 2차시 대부분 (API 키!) | 1차시 20분 | 1차시 15분 |
| 실습 시간 | 3차시 | 1차시 후반 + 2차시 | 1차시 후반 + 2차시 |
| 심화 활동 | 시간 부족 시 생략 | 2차시에 가능 | **배포 + Realtime 가능** |
| 회고 활동 | 없음 | Step 1 비교 | **3단계 전체 회고** |

> 💡 **교사 메모:**
> - Step 3는 **3단계 전체의 완결편**입니다. 단순히 Supabase를 배우는 것이 아니라, "내 컴퓨터에서만 되던 앱을 인터넷에 올려서 친구도 쓸 수 있게" 만드는 여정의 마무리입니다.
> - 2차시의 3단계 전체 회고가 매우 중요합니다. "왜 같은 앱을 3번이나 만들었는지"를 이해시키세요.
> - 답: 저장소만 바꾸면 앱의 성격이 완전히 달라진다 (로컬 → 클라우드 → 배포 가능). 이것이 소프트웨어 설계(아키텍처)의 핵심입니다.
> - Step 3를 마치면 학생들은 **실제 서비스와 유사한 구조**의 앱을 이해하게 됩니다. 이것이 이 교육 과정의 최종 목표입니다.
