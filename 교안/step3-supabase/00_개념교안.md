# 클라우드 데이터베이스 — 친구도 쓸 수 있는 앱 만들기

> **대상:** 중학교 1학년 ~ 고등학교 3학년 (수준별 표시 포함)

---

## 오늘 배울 내용

> Step 2에서 내 컴퓨터에 저장했던 카드 데이터를, 이번에는 **인터넷 어딘가에 있는 클라우드 DB**에 저장합니다. 드디어 친구도 접속할 수 있는 앱을 만들어 봅시다!

---

## 생각해보기

Step 2에서 우리는 카드를 `padlet.db` 파일에 저장했죠?

그런데 생각해보세요:

- Step 2에서 만든 `padlet.db` 파일을 친구에게 보내면 친구도 카드를 추가할 수 있을까요?
- **안 되죠!** 파일이 내 컴퓨터에만 있으니까.
- 친구가 카드를 추가해도, 그건 친구 컴퓨터에 있는 복사본일 뿐이에요.

**그럼 인터넷 어딘가에 DB를 두면 어떨까요?**

정답: **할 수 있습니다!** 그것이 바로 오늘 배울 **클라우드 데이터베이스**입니다.

---

## Step 1에서 배운 것 (빠르게 복습)

3단계 전체를 통해 이미 배운 개념입니다. 기억나나요?

- **데이터베이스(DB)** = 데이터를 정리해서 보관하는 장부
- **서버** = 요청을 받아서 처리하는 주방
- **API** = 주문서 양식 (GET, POST, PATCH, DELETE)
- **db.js** = 주방 도구 (바꿔 끼울 수 있는 부품!)

이 개념은 Step 3에서도 **그대로** 사용합니다. 바뀌는 것은 딱 하나, **데이터를 어디에 저장하느냐** 뿐입니다.

---

## 핵심 개념 1: 3단계 전체 여정

우리가 3단계에 걸쳐 해온 일을 한눈에 봅시다:

```
Step 1: 브라우저 → 서버 → Notion    (남의 서비스)
Step 2: 브라우저 → 서버 → padlet.db (내 컴퓨터)
Step 3: 브라우저 → 서버 → Supabase  (클라우드)
```

### 비유로 이해하기: 장부를 어디에 보관하느냐의 차이

| Step | 비유 | 설명 |
|------|------|------|
| Step 1 (Notion) | 남의 가게에 있는 장부 | 편하지만, 그 가게가 문 닫으면 못 봄 |
| Step 2 (SQLite) | 내 책상 위의 장부 | 내 맘대로 할 수 있지만, 친구가 못 봄 |
| Step 3 (Supabase) | 학교 서버실에 있는 장부 | 인터넷만 되면 어디서든 볼 수 있음! |

---

## 핵심 개념 2: 클라우드란?

**클라우드** = 인터넷 어딘가에 있는 컴퓨터

비유: 학교 서버실에 있는 컴퓨터를 생각해보세요.
- 서버실에 직접 가지 않아도, 와이파이만 되면 접속할 수 있죠?
- 집에서도, 카페에서도, 다른 학교에서도 접속 가능합니다.
- **클라우드**도 이것과 같습니다. 인터넷만 되면 어디서든 접속 가능!

```
Step 2 (SQLite)                      Step 3 (Supabase)
┌──────────┐                        ┌──────────────────┐
│ padlet.db │ ← 내 컴퓨터 안         │ Supabase 서버     │ ← 인터넷 어딘가
│ (파일 1개) │                       │ (클라우드 DB)      │
└──────────┘                        └──────────────────┘
   나만 접근 가능                      누구든 접근 가능!
```

---

## 핵심 개념 3: Supabase란?

**Supabase** = PostgreSQL을 쉽게 만들어주는 서비스

PostgreSQL은 전문 데이터베이스입니다. 그런데 직접 설치하면 **너무 어렵습니다.**

| 비교 | 직접 PostgreSQL 설치 | Supabase |
|------|---------------------|---------|
| 설치 | 복잡 (다운로드, 설정, 방화벽...) | 없음 (브라우저에서 클릭!) |
| 테이블 확인 | 까만 화면에 명령어 입력 | 웹 UI (Excel처럼 표로 보임) |
| API 자동 생성 | 직접 코딩해야 함 | 자동! |
| 무료 | 무료 | 무료 (프로젝트 2개까지) |
| 학생 경험 | "이거 너무 어려워요..." | "Notion이랑 비슷하네!" |

> **기억나나요?** Step 1에서 Notion 대시보드에서 데이터를 확인했죠?
> Supabase도 비슷합니다. 웹 대시보드에서 테이블을 눈으로 볼 수 있어요!

### 왜 Supabase?

직접 PostgreSQL을 설치하면:
1. 다운로드 + 설치 (10분)
2. 사용자 설정 (5분)
3. 데이터베이스 생성 (5분)
4. 외부 접속 설정 (10분)
5. 보안 설정 (10분)
6. ...아직도 끝이 안 남

Supabase를 쓰면:
1. 웹사이트 접속 + 로그인 (1분)
2. 프로젝트 만들기 (2분)
3. SQL 실행 (1분)
4. URL + Key 복사 (1분)
5. **끝!**

## 우리 앱 구조 비교

### Step 1 (Notion)

```
브라우저 (HTML/CSS/JS)
      │
      │ HTTP 요청 (fetch)
      ▼
Express 서버 (server.js)
      │
      │ Notion API 호출 (인터넷!)
      ▼
Notion 데이터베이스 (클라우드, 남의 서비스)
```

### Step 2 (SQLite)

```
브라우저 (HTML/CSS/JS)          ← 안 바뀜!
      │
      │ HTTP 요청 (fetch)       ← 안 바뀜!
      ▼
Express 서버 (server.js)        ← 안 바뀜!
      │
      │ SQL 명령 (로컬!)
      ▼
padlet.db 파일 (내 컴퓨터)
```

### Step 3 (Supabase)

```
브라우저 (HTML/CSS/JS)          ← 안 바뀜!
      │
      │ HTTP 요청 (fetch)       ← 안 바뀜!
      ▼
Express 서버 (server.js)        ← 안 바뀜!
      │
      │ Supabase SDK (인터넷!)
      ▼
Supabase (클라우드 PostgreSQL)   ← NEW!
```

### 중요한 발견!

> **server.js와 화면(public/)은 역시 안 바뀌었어요!**
>
> Step 1에서 Step 2로 넘어갈 때도, Step 2에서 Step 3으로 넘어갈 때도
> 바뀐 것은 **db.js** 파일 하나뿐입니다.
>
> 3번이나 저장소를 바꿨는데, server.js와 화면은 한 글자도 안 바뀌었습니다.
> 이것이 **좋은 설계(모듈화)**의 힘입니다!

## 이번 단계에서 바뀐 것

| 파일 | 변경 내용 |
|------|---------|
| **db.js** | SQLite(better-sqlite3) → Supabase(@supabase/supabase-js) (**이것만 바뀌었어요!**) |
| **package.json** | `better-sqlite3` 빠지고 `@supabase/supabase-js` 추가 |
| **.env** | Supabase URL + anon key 추가 |
| server.js | 변경 없음 |
| public/ | 변경 없음 |

## 3단계 전체 비교표

| | Step 1 (Notion) | Step 2 (SQLite) | Step 3 (Supabase) |
|--|----------------|----------------|-----------------|
| DB 위치 | Notion 서버 | 내 컴퓨터 파일 | 클라우드 서버 |
| 데이터 확인 | Notion 앱 | VS Code / SQLite Viewer | Supabase 대시보드 |
| 동시 접속 | 가능 | 불가 (1명) | 가능 (여러 명) |
| 인터넷 없이 | 불가 | 가능 | 불가 |
| 배포 가능 | 어려움 | 어려움 | 쉬움 (Vercel) |
| SQL 사용 | 안 함 | 직접 작성 | 자동 생성 |
| API 키 필요? | 필요 (복잡!) | 불필요 | 필요 (간단) |
| .env 설정 | API 키 + DB ID | 거의 불필요 | URL + Key |
| 데이터 지속성 | 서버 꺼도 유지 | 서버 꺼도 유지 | 서버 꺼도 유지 |
| 실제 서비스 느낌 | 중간 | 낮음 | **가장 높음!** |
