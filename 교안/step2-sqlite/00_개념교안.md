# 내 컴퓨터에 데이터 저장하기 — SQL과 SQLite 입문

> **대상:** 중학교 1학년 ~ 고등학교 3학년 (수준별 표시 포함)

---

## 오늘 배울 내용

> Step 1에서 Notion에 저장했던 카드 데이터를, 이번에는 **내 컴퓨터 파일 하나**에 저장합니다. 인터넷 없이도 동작하는 앱을 만들어 봅시다!

---

## 생각해보기

Step 1에서 우리는 카드를 추가하면 **Notion**에 데이터가 저장되었죠?

그런데 생각해보세요:

- Notion 서버가 고장 나면 우리 앱은 어떻게 될까요?
- 인터넷이 안 되는 곳에서도 카드를 저장할 수 있을까요?
- 데이터를 내 컴퓨터에 직접 저장하면 안 될까요?

정답: **할 수 있습니다!** 그것도 파일 **딱 하나**로요.

오늘은 Notion 대신 **SQLite**라는 도구를 써서, 내 컴퓨터에 데이터를 저장하는 방법을 배워봅니다.

---

## Step 1에서 배운 것 (빠르게 복습)

Step 1에서 이미 배운 개념입니다. 기억나나요?

- **데이터베이스(DB)** = 데이터를 정리해서 보관하는 장부
- **서버** = 요청을 받아서 처리하는 주방
- **API** = 주문서 양식 (GET, POST, PATCH, DELETE)

이 개념은 Step 2에서도 **그대로** 사용합니다. 바뀌는 것은 딱 하나, **데이터를 어디에 저장하느냐** 뿐입니다.

---

## 핵심 개념 1: SQLite — 파일 하나가 곧 데이터베이스

```
Step 1 (Notion)                    Step 2 (SQLite)
┌──────────┐                      ┌──────────┐
│ Notion    │ ← 인터넷 필요       │ padlet.db │ ← 내 컴퓨터 파일
│ (클라우드) │                     │ (파일 1개) │
└──────────┘                      └──────────┘
   멀리 있음                          바로 옆에 있음
```

### 비유로 이해하기: 구글 시트 vs 엑셀 파일

- **Notion** = 구글 시트 (인터넷에 있는 장부, 인터넷 필요)
- **SQLite** = 엑셀 파일 (내 컴퓨터에 있는 장부, 인터넷 불필요!)

SQLite는 `padlet.db`라는 **파일 하나**에 모든 데이터를 저장합니다.
이 파일이 곧 데이터베이스입니다. 정말 간단하죠?

---

## 핵심 개념 2: SQL — 데이터베이스에게 말하는 언어

**SQL**(Structured Query Language)은 데이터베이스에게 "이거 해줘"라고 말하는 **언어**입니다.

### 비유로 이해하기: 도서관 사서에게 부탁하기

도서관에서 책을 찾고 싶으면, 사서에게 이렇게 말하죠:

| 하고 싶은 일 | 사서에게 하는 말 | SQL로 하는 말 |
|------------|---------------|-------------|
| 장부 만들기 | "새 대출 장부 만들어주세요" | `CREATE TABLE cards (...)` |
| 찾아보기 | "수학 관련 책 목록 알려주세요" | `SELECT * FROM cards` |
| 추가하기 | "이 책 대출 기록 추가해주세요" | `INSERT INTO cards (...)` |
| 삭제하기 | "이 대출 기록 지워주세요" | `DELETE FROM cards WHERE id = ?` |

SQL 문법 4가지만 알면 대부분의 데이터 작업을 할 수 있습니다!

### SQL의 장점: 내가 직접 장부를 설계한다

Step 1에서 Notion DB를 만들 때, 컬럼 이름의 대소문자(color vs Color) 때문에 문제가 생긴 적 있었죠?

SQL에서는 **내가 직접** 장부를 만들기 때문에 이런 문제가 없습니다:

```sql
CREATE TABLE cards (
    id         TEXT PRIMARY KEY,
    title      TEXT NOT NULL,
    content    TEXT DEFAULT '',
    color      TEXT DEFAULT 'yellow'
);
```

이렇게 정확하게 정의해두면, 대소문자 혼동이 생길 일이 없습니다.

## Notion DB vs SQLite 비교

| 항목 | Step 1: Notion | Step 2: SQLite |
|------|---------------|---------------|
| 데이터 저장 위치 | Notion 서버 (클라우드) | 내 컴퓨터 (padlet.db 파일) |
| 인터넷 필요? | 필요 | **불필요!** |
| API 키 필요? | 필요 (발급 과정 복잡) | **불필요! 훨씬 쉽다!** |
| .env 설정 | API 키, DB ID 필수 | 거의 필요 없음 |
| 데이터 확인 방법 | Notion 웹사이트 접속 | VS Code에서 바로 열기 |
| 데이터 언어 | Notion API (자체 형식) | **SQL (표준 언어)** |
| 속도 | 인터넷 속도에 따라 다름 | **매우 빠름** (로컬 파일) |
| 서버 꺼도 데이터 유지? | O (클라우드 저장) | **O (파일 저장)** |

## 우리 앱 구조 비교

### Step 1 (Notion)

```
브라우저 (HTML/CSS/JS)
      │
      │ HTTP 요청 (fetch)
      ▼
Express 서버 (server.js)
      │
      │ Notion API 호출 (인터넷!)
      ▼
Notion 데이터베이스 (클라우드, 멀리 있음)
```

### Step 2 (SQLite)

```
브라우저 (HTML/CSS/JS)          ← 안 바뀜!
      │
      │ HTTP 요청 (fetch)       ← 안 바뀜!
      ▼
Express 서버 (server.js)        ← 안 바뀜!
      │
      │ SQL 명령 (로컬!)
      ▼
padlet.db 파일 (내 컴퓨터)       ← NEW!
```

### 중요한 발견!

> **server.js와 화면(public/)은 안 바뀌었어요!**
>
> 바뀐 것은 **db.js** 파일 하나뿐입니다.
> 데이터를 가져오는 "도구"만 바꿨을 뿐, 주문을 받는 방식은 그대로입니다.

이것이 바로 **좋은 설계**입니다. 저장 방식을 바꿔도, 나머지 코드에 영향을 주지 않습니다.
